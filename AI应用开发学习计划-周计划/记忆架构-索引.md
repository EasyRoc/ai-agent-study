# 记忆架构（Memory Architecture）索引

> 与 [`../AI应用开发学习计划-Java与Python双轨.md`](../AI应用开发学习计划-Java与Python双轨.md) 中**记忆架构**子节、各周 `md` 配套。面试时常称 **「对话记忆 / 长短期记忆 / 与 RAG 关系」**。

## 1. 为什么要单独谈「记忆架构」

大模型**本身**的「记忆」只有 **上下文窗口** 内的 token；**产品级** 记忆=你在**应用层** 显式设计：**存什么、按谁分桶、何时读、何时写、如何删、花多少 token**。  
本计划不引入过多框架，**以分层 + 能画图 + 能写表结构** 为达标线。

## 2. 分层速查（本计划采用的划分）

| 层次 | 常叫法 | 存什么 | 典型实现（本计划） | 主要落在第几周 |
|------|--------|--------|--------------------|----------------|
| **L0 工作/短期记忆** | 对话上下文、Session | 多轮 `messages`、**滑窗/截断** 后的子集 | 内存/Redis/DB 的 `thread_id` 维度消息表 | **2** 必会；**6** 与 Agent 多轮**结合** |
| **L1 摘要记忆** | Rolling summary、Memory summary | 历史过长时，**用模型** 把旧对话压成 1 段 `summary` 再进上下文 | 周期性 LLM 调用 或 每 N 轮一次 | **2** 选做；**5** 可深化 |
| **L2 外显/检索记忆** | 企业知识、文档记忆 | 非参数量化知识：**chunk+向量** 或 BM25，**不**进权重 | RAG/向量库，**和「对话事实」** 分库存 | **3～4** 核心；**5** 混合检索 |
| **L3 长期/用户事实（Semantic / Profile）** | 用户画像、长期偏好、可积累事实 | 「我叫什么」「常问哪个项目」等**可结构化** 或 **可嵌入** 片段 | 表或向量库，**`userId`+TTL** 强约束 | **5** 概念+轻量 demo；**6** 可用 **Tool 写入** 选做 |
| **L4 治理与隔离** | 多租户、合规 | **谁** 的记忆与**谁** 隔离、**可删除**、**留存策略** | `tenantId` / `userId` 所有读写路径；**第 8 周** 与隐私/审计 对齐 | **5～6** 设计意识；**8** 必谈 |

> **注意**：**RAG（L2）** 是「**知识在文档里**」；**L3** 是「**关于** **用户/会话** 的**可写**事实」—— 两者**常并存**，**表结构不要** 混成一张「啥都塞向量」。

## 3. 与其他「流行技术」关系

- **RAG** ≈ 给模型接 **L2 外显记忆**（**读多**、**可引用** 文档片段）。
- **Function Calling** ≈ 模型**主动** 触发**写/读** L3 或 外部系统（`save_user_pref`、`query_crm`）。
- **MCP** ≈ 工具/资源 的**暴露**方式；**记忆库** 也可作为 **MCP resource 或 tool**（**选读**，本计划不强制第 6 周必做记忆 MCP）。

## 4. 反模式（面试能说出来即可）

- 把**整段** 对话历史**无截断** 一直塞进 prompt → **费钱**、**越界**、**混乱**。
- 把**用户隐私** 和 **公司文档** 进**同一** 向量 collection **且** 不做 **元数据** 隔离 → 检索**串** 租户。
- 以为「用了向量库」= 有「长期记忆」→ 需说清是 **L2 知识** 还是 **L3 用户事实**。

## 5. 可选生态（不强制在计划内实装）

- **LangChain** `ChatMessageHistory` / 对话缓冲；**Mem0、Zep、Letta** 等 **记忆即服务** —— 了解**产品定位**即可，**你的 Java 网关上** 仍要 **sessionId、持久化、审计**。

## 6. 建议产出物（跨周汇成 1 份）

  - 一张 **Mermaid/ASCII** 图：用户请求 → L0 组装上下文 → 可选 L1 摘要 → 可选 L2 检索 → 模型；
  - 小表：各层 **存哪、谁 key、保留多久**。
