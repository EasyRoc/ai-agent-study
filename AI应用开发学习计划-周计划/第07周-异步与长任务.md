# 第 7 周：异步、削峰与长任务

> 约 **5～7 个学习日**。目标：**长任务**从提交到完成状态可追踪；优先 **Kafka + Java** 与你经验对齐；Python 可用 **Celery/Redis/简易 Thread** 做对照理解。

## 本周目标

- 场景：**生成一份长报告** 或 **批量 embedding 重建索引**，用户不能同步等 2 分钟。
- 提供：`POST /jobs` → `202` + `jobId`；`GET /jobs/{id}` → `PENDING|RUNNING|SUCCESS|FAIL` + `result` 或 `error`。
- **重试 1 次**策略：仅 `FAIL` 且错误类型为**可重试**（如超时）时，手动或管理接口触发（**演示**即可，不必全自动化）。

## 按日计划

### 第 1 天：状态与表结构

- [ ] 设计 `job` 实体：`id`, `type`, `status`, `createdAt`, `updatedAt`, `payload`, `result`, `errorCode`（可用内存/ H2/ MySQL 任一）。
- [ ] 写出状态机**合法转移**：`PENDING→RUNNING→SUCCESS|FAIL`。

### 第 2～3 天：Java + Kafka 路径（推荐主路径）

- [ ] 定义 topic：`ai.jobs`（名可自拟）；message：`jobId` + 必要 `payload`。
- [ ] `POST /jobs` 写库 `PENDING`，发 Kafka，立即返回 `jobId`。
- [ ] `@KafkaListener` 消费：改 `RUNNING` → 调**慢操作**（可 Thread.sleep+mock 或 真调你 Python `/heavy`）→ `SUCCESS`/`FAIL`。
- [ ] `GET /jobs/{id}` 查库。

### 第 4 天：Python 对照（二选一或简化）

- [ ] 方案 A：`FastAPI` 背景任务 `BackgroundTasks` 仅**内存 job**（进程重启丢），**理解**用。
- [ ] 方案 B：Redis 队列 + worker 进程（有精力再做）。

### 第 5 天：重试与幂等

- [ ] 消费失败时 `FAIL` + 记录堆栈**摘要**（勿记敏感 payload）。
- [ ] 提供 `POST /jobs/{id}/retry`：仅 `FAIL` 可转，再发一条 Kafka 或同步改 `PENDING` 再发（**演示**）。

### 第 6 天：与「模型调用」结合

- [ ] 长任务内部真实调用**一次** RAG/Agent（可缩短生成长度以省 token）。
- [ ] 记录 **任务起止** 与 **模型总耗时** 到 `result` 的 JSON 里。

### 第 7 天：图与说明

- [ ] 画**状态图** + **时序图**（可 Mermaid），放入 `docs/job-flow.md`。
- [ ] 在 `README` 写：`curl` 创建 job、轮询直到 `SUCCESS` 的 **shell 小脚本** 片段。

## 周产出物

- [ ] `docs/job-flow.md`（两图 + 3 条 bullet 与第 4～9 周项目关系）。
- [ ] 可运行 demo：至少**一条**长任务全链路（Kafka 或纯内存后台二选一，**在 README 标明**）。

## 完成标准

- 能解释：**为什么** RAG/长生成不适合纯同步、**什么** 时候必须 `202 + 轮询` 或 WebSocket 推送（面试口径）。

## 与第 8 周衔接

任务完成时**顺便**把 **input/output tokens 累计** 写入，为成本统计埋伏笔。

**与记忆**（见 [`记忆架构-索引`](./记忆架构-索引.md)）：在 `docs/job-flow.md` 中**标注** 哪类 job 会**改变 L2 索引**（ingest、re-embed）或**触发 L1 摘要**；避免「排队」与业务**无关联** 的空话。
