## 1. 关系一句话

| 概念 | 一句话 |
|------|--------|
| **Function Calling**（**Tool / Tools**） | 模型在**一次 API 返回**中给出**结构化**的「要调哪个工具、参数是什么」；**宿主**执行后把**结果回注**再模型续写。多数云厂商的**原生**能力。 |
| **ReAct** | **Reasoning + Acting**：**思考 → 行动（调工具）→ 观察** 的多轮循环，可用 **FC 在底层**实现，也可用**纯文本**在 prompt 里自拼（工程上**少用纯文本**）。 |
| **MCP**（Model Context Protocol） | **标准协议**，把**工具（Tools）、资源（Resources）、提示（Prompts）** 以 **MCP 服务端** 形式暴露，**MCP 客户端**（IDE、自研 Agent、网关）**发现**并**安全调用**；解决「每个产品各造一套工具对接」的碎片化。 |
| **Skills**（**技能包**，不同产品叫法不同） | **可版本化、可发现** 的**能力模块**：**何时启用**、**分步 SOP**、**依赖哪些工具/数据** 写进一份**声明**（如 `SKILL.md`）；**不等价于**单个 HTTP 接口，更像**产品级编排+文档契约**。 |
| 与 RAG 关系 | RAG 常作为 **MCP/Agent 的其中一个工具** 或 **Skill 的一步**；**检索** 不是 ReAct/FC 的替代，是**可组合能力**。 |
| **记忆架构** | **L0 会话** / **L1 摘要** / **L2 检索 RAG** / **L3 用户事实** / **L4 治理** 分层，见 [**记忆架构-索引**](./记忆架构-索引.md)；与 **RAG（多指 L2）**、**多轮（L0）** 面试常一起问。 |

## 3. 面试常问（可自查）

- **FC 和 ReAct 是不是一回事？**  
  — **不是**：ReAct 是**推理范式/循环**；FC 是**单轮 API 的调用形态**；实现 ReAct 时常用多轮 **FC**。
- **为什么搞 MCP 而不是再写一堆 REST？**  
  — **元数据**（tools list、schema）、**鉴权/传输**（stdio、SSE 等）**可插拔**、**生态**中 IDE/Agent 能复用**同一**服务端；REST 也成立，MCP 是**约定大于配置** 的一类方案。
- **Skill 和 Tool 差在哪？**  
  — **Tool** 是**一个**函数级能力；**Skill** 是**何时用、用哪几个、顺序与边界** 的**成包说明**，常含**人可读**的 SOP 与**机器可读** 的元数据。

## 4. 与厂商名词对照（避免混淆）

- **OpenAI / 兼容端**：`tools` + `tool_choice`，返回 `tool_calls`。  
- **Anthropic / 部分国产**：`tools` 或等价字段，名称可能叫 **Tool use**。  
- **本计划** 统一写 **Function Calling = 原生的「模型选工具+参数」**；**自写**「以 JSON 输出 action」算 **ReAct 风格**但**不**计为同等的工程实践（除非**对比实验**用）。

## 5. 延伸（本计划不强制）

- **A2A / 多 Agent 协议**、**MCP 与 OAuth** 细粒度、**MCP 资源**（`resources/read`）与 **RAG 文档** 的边界；感兴趣可在第 9 周后**自选**补读官方规范。
